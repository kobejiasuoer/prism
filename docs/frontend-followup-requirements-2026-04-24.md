# Prism 新前端补改需求

日期：2026-04-24

基于当前 `apps/web` 新前端与老版 `apps/control-panel` 的对照梳理。

## 需求背景

Prism 已经有一套新的 `Next.js` 前端骨架，页面结构、视觉风格、统一股票页、移动端适配都已经明显优于老版控制台。

但当前版本仍然更像“新决策前台”，还不能完整替代老版 `FastAPI + Jinja` 控制台。主要缺口集中在后台操作层和闭环能力上：

1. 问股闭环没有完成
2. 持仓管理动作没有迁完
3. 参数编辑没有迁完
4. 任务触发和日志查看没有迁完
5. 深层证据入口和刷新控制没有迁完

这次补改的目标不是再做一轮重设计，而是把现有新前端补齐到“可以作为默认操作入口”的程度。

## 总目标

让 `apps/web` 成为 Prism 的默认前端入口，满足日常使用时不必频繁回退到老版 `apps/control-panel` 页面。

## 核心原则

1. 不重写后端数据逻辑，尽量复用现有 FastAPI API。
2. 不推翻现有新前端视觉风格，沿用 `apps/web` 当前设计系统和组件体系。
3. 优先补齐闭环能力，而不是继续扩页面数量。
4. 优先做“能操作”的页面，而不是只做“能展示”的页面。

## 本次必须完成的范围

### P0. 问股闭环补齐

当前问题：

- 新前端已经有全局搜索和统一股票页
- 但旧版 `ask` 页面最关键的“追问 followup”还没有迁过去
- 最近查询、快速示例、页内股票管理动作也没迁过去

需求：

1. 在新前端补齐问股闭环，不再依赖旧 `/ask` 页面完成深度单股分析。
2. 在 `/stock/[code]` 内增加独立的“追问”能力。
3. 支持调用现有接口：
   - `GET /api/ask?q=`
   - `GET /api/ask/suggest?q=`
   - `POST /api/ask/followup`
4. 追问区需要保留对话上下文，至少支持连续追问。
5. 需要保留“从搜索进入个股页后直接继续追问”的体验，不要让用户跳回旧页面。
6. 如果旧版问股页里有“加入持仓/恢复持仓”能力，也迁到新前端的股票页里。

验收标准：

- 用户可以只在新前端完成“搜股票 -> 看结论 -> 继续追问 -> 回看证据”整条链路。
- 日常问股不再需要打开旧 `/ask`。

### P0. 持仓管理动作补齐

当前问题：

- 新版 `/portfolio` 已经能看持仓分组
- 但还不能完整做名单管理和刷新跟踪

需求：

1. 在 `/portfolio` 页面补齐持仓管理区。
2. 支持现有接口：
   - `POST /api/watchlist/manage/add`
   - `POST /api/watchlist/manage/archive`
   - `POST /api/watchlist/manage/restore`
3. 支持展示当前活跃持仓池和归档列表。
4. 支持添加、归档、恢复后的即时反馈。
5. 尽量补上刷新状态、步骤进度、相关日志入口。

验收标准：

- 用户在新前端可完成持仓名单的增删改恢复。
- 不需要再回到旧 `watchlist` 页面做基础管理。

### P0. 参数编辑补齐

当前问题：

- 新版 `/settings` 页面只有状态面板
- 参数 API 已经封装，但前端没有实际编辑能力

需求：

1. 在 `/settings` 下补齐参数编辑模块。
2. 支持现有接口：
   - `GET /api/parameters`
   - `POST /api/parameters`
3. 页面至少要有：
   - 参数摘要
   - 当前读写文件路径
   - 必填阈值组提示
   - 原始 JSON 编辑器
   - 重新加载磁盘版本
   - 保存并显示校验结果
4. 编辑器不一定必须上 Monaco，但体验不能退回成难用的纯文本块。
5. 保存失败时要给出明确错误提示，不能静默失败。

验收标准：

- 用户可以只在新前端完成参数查看、编辑、保存和错误修正。
- 不再依赖旧 `/parameters` 页面。

### P0. 任务运行与日志查看补齐

当前问题：

- 新版 `/settings` 只展示任务和运行记录
- 还不能真正执行任务，也没有日志查看闭环

需求：

1. 在 `/settings` 补齐任务操作能力。
2. 支持现有接口：
   - `GET /api/overview`
   - `GET /api/runs`
   - `POST /api/tasks/{task_name}/run`
   - `GET /api/runs/{run_id}`
   - `GET /api/runs/{run_id}/log`
   - `GET /api/preview`
3. 支持任务启动、状态刷新、最近运行记录、日志查看。
4. 如果旧版有“允许发飞书”的参数，也迁到新前端。
5. 日志和预览不要只做下载链接，最好有页内 drawer / modal 预览。

验收标准：

- 用户可以在新前端完成常用任务的启动、状态确认和日志查看。
- 旧控制台只作为兜底，不再是主操作入口。

### P0. 证据入口与刷新控制补齐

当前问题：

- 新前端已经有来源卡片，但旧版那种“刷新状态条 + 手动触发刷新 + 证据回源入口”没有完整迁移

需求：

1. 为首页、持仓页、观察池页、复盘页、个股页补齐统一的证据与刷新组件。
2. 支持现有接口：
   - `GET /api/refresh/status?page=`
   - `POST /api/refresh/trigger`
   - `GET /api/preview`
   - `GET /artifacts`
3. 每个核心页面至少能做到：
   - 看见数据是否新鲜
   - 主动刷新
   - 打开关键 artifacts / logs / quality json
4. 这些能力建议做成可复用组件，不要每页各写一套。

验收标准：

- 用户在新前端内能完成“看结论 -> 核对来源 -> 打开原始证据 -> 手动刷新”的闭环。

## 第二优先级范围

### P1. 旧 detail / batch 页能力回收

需求：

1. 评估是否需要恢复独立 detail 页面。
2. 如果不恢复独立页面，也要保证旧 detail 页的关键信息在新单页中能完整找到。
3. 尤其要补齐：
   - 观察池批次细节
   - 复盘 detail 细节
   - 个股历史变化视角

### P1. 可信度开关补齐

需求：

1. 老版 `review` 和 `opportunities` 的可信度开关如果仍有业务价值，需要迁回新前端。
2. 不要只迁视觉标签，要迁可操作逻辑和状态解释。

### P1. 搜索体验补齐

需求：

1. 把旧 `ask` 页的最近查询、示例候选、快捷进入体验补进新 `Command Bar` 或股票页。
2. 保证搜索不是只有联想结果，还能形成可连续使用的入口。

## 明确不要求本次做的事

1. 不要求重写 `dashboard_data.py`。
2. 不要求重做一套新的 UI 风格。
3. 不要求把老版所有页面 1:1 原样照搬。
4. 不要求新增后端业务逻辑，除非现有 API 确实缺少必要字段。

## 技术要求

1. 前端继续使用现有 `apps/web` 技术栈：
   - Next.js
   - React
   - TypeScript
   - TanStack Query
   - 当前组件体系
2. 继续复用当前路由结构：
   - `/`
   - `/portfolio`
   - `/discovery`
   - `/review`
   - `/settings`
   - `/stock/[code]`
3. 继续保留当前 rewrite/redirect 方案，不要破坏现有 API 代理。
4. 需要保证桌面端和移动端都能正常使用。
5. 需要保留当前键盘优先体验，尤其是全局搜索。

## 完成定义

以下条件全部满足，才算这轮补改完成：

1. 日常主流程不再依赖旧 `apps/control-panel` 页面。
2. 新前端可以独立完成：
   - 今日决策查看与动作标记
   - 单股搜索、结论查看和追问
   - 持仓名单管理
   - 参数编辑与保存
   - 常用任务运行与日志查看
   - 来源校验、回源和刷新
3. 老控制台只保留为兜底入口，而不是日常必开页面。

## 最后一句话

这次补改的核心，不是“再做更漂亮一点”，而是把新前端从“展示层”补成“真正可操作的默认入口”。
