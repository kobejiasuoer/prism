# Prism 问股页设计方案 v0

日期：2026-04-19
状态：设计草案，可直接进入 Phase 1 开发
范围：`workspace-invest/skills/invest-flow/control_panel`

## 1. 这页为什么要做

现有 Prism 已经有四个真实产品面：

- `Today`：今天先看什么
- `Watchlist`：已关注 / 已持仓今天怎么处理
- `Opportunities`：今天有哪些新机会
- `Review`：历史回看与环境校正

但还缺一个高频入口：

> “给你一只股票，你现在就告诉我，这票值不值得看、今天该怎么处理、为什么。”

这不是 `Watchlist`，因为用户可能问任意股票；
这也不是 `Opportunities`，因为用户问的票不一定在今天候选池里；
它本质上是一个“单票统一诊断入口”。

结论：

- 新增独立产品面 `Ask`
- URL 建议：`/ask`
- 导航文案建议：`问股`

## 2. 这页要回答的核心问题

问股页不应该输出一大坨信息，而要先回答 3 个问题：

1. 这只股票今天处于什么状态？
2. 为什么是这个判断？
3. 如果我要跟，它的触发位 / 风险位 / 失效位分别是什么？

所以页面必须坚持 Prism 现在已经形成的原则：

- 先给动作，再给依据
- 先收束结论，再展开细节
- 原始文件可回看，但不让原始 JSON 直接变主界面

## 3. 产品定位

### 3.1 和现有产品面的关系

- `Today`：今日总控，负责排序
- `Watchlist`：固定自选 / 持仓判断
- `Opportunities`：新机会发现
- `Ask`：任意单票即时诊断
- `Review`：历史研究与环境校验
- `Ops`：运行、质检、日志

### 3.2 Ask 页的独特价值

它不是“把 Watchlist 详情页复制一份”。

它的价值是把一只股票在多个系统里的状态收束到一个页面里：

- 如果它已在自选股里，直接继承 `watchlist_snapshot`
- 如果它命中过机会池，直接继承 `candidate`
- 如果它进入午盘确认，补上 `confirmation`
- 如果它从未进入现有链路，也能即时按需抓取并给出临时结论

所以 Ask 页本质上是：

> `单票聚合页 + 按需分析页`

## 4. 页面结构

## 4.1 顶部 Hero

Hero 不是一句介绍词，而是一个“可以立刻输入股票”的工作区。

应包含：

- 股票输入框：支持代码 / 名称
- 市场自动识别：沪 / 深 / 北
- 主按钮：`开始分析`
- 次按钮：`查看最近问过`
- 快捷示例：最近 5 只、当前自选股、今天候选股

Hero 下方直接显示本次分析状态：

- `已命中自选股`
- `已命中机会池`
- `仅临时分析`
- `数据拉取中`
- `数据不完整`

## 4.2 第一屏：一页先给结论

第一屏只保留最关键的 4 块：

1. `今日结论`
   - 回避 / 观望 / 轻仓跟踪 / 可做 / 只观察不执行
2. `核心原因`
   - 一句话总结，例如“技术修复未获资金确认”
3. `执行边界`
   - 触发位 / 回踩位 / 失效位 / 止损位
4. `置信度`
   - 高 / 中 / 低
   - 并明确原因：数据齐全、一致性高，或数据缺失、事件未验证

这部分要能做到：

> 用户只看第一屏，也知道今天该不该动。

## 4.3 第二屏：为什么这样判断

建议拆成 6 张组卡，而不是写成长文：

1. 技术面
   - 技术基调
   - 规则分 / calc_score
   - 均线位置
   - 量价结构
2. 资金面
   - 主力流向
   - 5 日资金趋势
   - 是否有资金确认
3. 事件面
   - 最新公告
   - 最新新闻
   - 事件偏向
4. 基本面
   - PE / PB / ROE / 市值
   - 估值与盈利是否匹配
5. 情绪 / 板块
   - 所属板块
   - 主线位置
   - 是否强于行业
6. 风险
   - 硬风险
   - 追高风险
   - 数据缺失风险

每组卡只保留：

- 结论标签
- 2-4 条 bullet
- 一个最重要的补充指标

## 4.4 第三屏：跨系统状态

这块是 Ask 页最关键的差异化模块。

需要明确展示：

- 自选股状态：是否在 `Watchlist`
- 机会池状态：是否在今日 `Opportunities`
- 午盘状态：是否进入 `Confirmation`
- 今日总控状态：是否出现在 `Today` 优先队列
- 历史状态：最近一次出现在哪条链路

目的：

- 避免用户问一只票时，只看到单次抓取结果
- 直接告诉他这只票在系统里“到底有没有被真正看上”

## 4.5 第四屏：行动计划

这一屏建议固定成右侧或中下方行动卡，结构统一：

- `动作`
- `触发`
- `回避`
- `失效`
- `仓位`

如果是强结论票，再补：

- `如果持有`
- `如果未持有`

这样 Ask 页能同时服务两种问题：

- “我持有它怎么办？”
- “我没持有，现在能不能做？”

## 4.6 第五屏：证据与原始文件

保留 Prism 现有的“可回看原始文件”能力，但只放在页尾：

- 抓取快照 JSON
- 自选股快照 JSON
- 早盘批次 JSON
- 午盘确认 JSON
- 原始公告 / 新闻列表

## 5. 页面视觉建议

延续现有 Prism 深色体系，不另起风格。

建议方向：

- 保留深色背景、绿/蓝/琥珀状态色
- Hero 输入区做成“工作台”而不是普通表单
- 第一屏结论卡要明显大于其他卡片
- `结论` 与 `执行边界` 用更高对比度模块
- `为什么` 区域保持规则化卡片，不做聊天气泡

不建议：

- 做成纯聊天界面
- 第一屏塞满新闻流
- 把原始 JSON 或长段摘要直接放到顶部

## 6. 交互流程

建议交互如下：

1. 用户输入股票代码 / 名称
2. 前端先做代码解析与最近记录命中
3. 后端立即返回第一层上下文：
   - 是否命中 `Watchlist`
   - 是否命中 `Opportunities`
   - 是否已有缓存结果
4. 同时启动问股分析任务
5. 页面先显示 skeleton + 阶段状态
6. 分析完成后按模块逐块渲染

阶段状态建议固定 4 步：

1. 解析股票身份
2. 读取系统内已有上下文
3. 拉取实时行情 / 资金 / 公告 / 新闻
4. 生成统一判断

## 7. 数据与实现

## 7.1 现有能力可直接复用

当前系统已经有以下能力，不需要重做：

- `watchlist_snapshot`
- `candidate`
- `confirmation`
- `today-brief`
- `stock-analyzer/scripts/fetch.py --codes ...`
- `fetch.py` 内的公告 / 新闻 / 资金 / 技术 / 基本面抓取

因此 Ask 页的后端不应该重新写一套采集器，而应该做：

- 统一聚合
- 按需补抓
- 统一渲染

## 7.2 新 canonical object 建议

新增实体：`stock_case`

用途：

- 表示“一次针对某只股票的统一诊断结果”

建议字段：

- `case_id`
- `generated_at`
- `trade_date`
- `query`
- `stock`
  - `code`
  - `name`
  - `market`
- `source_context`
  - `in_watchlist`
  - `in_screening_batch`
  - `in_confirmation`
  - `in_today_queue`
- `market_snapshot`
  - `price`
  - `change_pct`
  - `amount`
  - `industry`
- `rule_snapshot`
  - `tech_base`
  - `flow_base`
  - `event_base`
  - `score`
- `fundamentals`
- `capital_flow`
- `events`
  - `announcements`
  - `news`
- `decision`
  - `label`
  - `summary`
  - `confidence`
- `action_plan`
  - `action`
  - `trigger`
  - `avoid`
  - `invalidate`
  - `sizing`
  - `levels`
- `risk_flags`
- `artifacts`

## 7.3 路由建议

- HTML：
  - `/ask`
- API：
  - `POST /api/ask/analyze`
  - `GET /api/ask/jobs/{job_id}`
  - `GET /api/ask/latest?code=600036`

如果想先做最小版本，也可以先上同步接口：

- `GET /api/ask?code=600036`

但中期更建议走异步，因为公告 / 新闻 / 资金抓取会拉长首屏时间。

## 7.4 后端拼装顺序

推荐顺序：

1. `resolve_stock_identity(query)`
2. `load_watchlist_snapshot(code=...)`
3. `find_candidate_detail(code=...)`
4. `load_confirmation()` 并匹配 code
5. 如果已有数据不足：
   - 调 `stock-analyzer/scripts/fetch.py --codes CODE --skip-snapshot-save`
6. 统一生成 `stock_case`
7. 页面模板只消费 `stock_case`

关键原则：

- 模板层不直接碰零散脚本输出
- 页面只吃 canonical object
- 原始脚本继续负责数据获取，不负责页面语义

## 8. 开发分期

### Phase 1：低风险可落地版

目标：

- 先把问股页跑起来
- 先支持代码输入
- 先聚合已有系统数据 + `fetch.py --codes`

包含：

- `/ask` 页面
- `/api/ask?code=...`
- 基础 Hero
- 结论卡
- 六维理由卡
- 跨系统状态卡
- 原始文件入口

不做：

- 自然语言追问
- 多轮问答
- 历史问股记录管理

### Phase 2：体验增强版

加入：

- 名称搜索与候选下拉
- 最近问过 / 收藏 / 一键加入自选股
- 异步任务与加载状态
- 最近 3 次问股结论对比

### Phase 3：真正的“问股 Copilot”

加入：

- 基于 `stock_case` 的二次追问
- 例如：
  - “如果我已经持有怎么办？”
  - “它和同板块哪个更强？”
  - “今天为什么不是买点？”

## 9. 我建议的最终产品形态

导航建议改为：

- `今日`
- `问股`
- `自选股`
- `机会池`
- `复盘`
- `控制台`

原因：

- `问股` 是高频单点入口，应该比 `复盘` 更靠前
- 它连接了 `自选股` 和 `机会池`
- 它会成为最自然的外部演示页面

一句话总结：

> `Today` 回答“今天先看谁”，`Ask` 回答“这只票到底行不行”。

## 11. 推荐结论

推荐直接做。

而且建议不要把它当成“详情页的另一个变体”，而是把它定义成 Prism 的第五个真实产品面。

这会让 Prism 的产品结构变得完整：

- 总控看 `Today`
- 单票诊断看 `Ask`
- 已关注看 `Watchlist`
- 新机会看 `Opportunities`
- 历史验证看 `Review`
